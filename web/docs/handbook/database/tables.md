# Database — Tables (full reference)

This is the complete, column-by-column reference for every table in WorldNews-101. Everything here was read directly from the migration files in `/home/jiwira/Projects/WorldNews-101/db/migrations/`:

- `0001_init.sql` — `stories`, `articles`, extensions, indexes
- `0002_sources.sql` — `articles.author`, `sources`, `briefings`
- `0003_translations.sql` — `translations` jsonb on `stories` and `briefings`

There are **four** tables: `stories`, `articles`, `briefings`, `sources`. (There is **no** `questions` table — see the note in [../02-database.md](../02-database.md).)

A few PostgreSQL type reminders for newcomers:

- `uuid` — a 128-bit unique id, here generated by `gen_random_uuid()` (from the `pgcrypto` extension).
- `timestamptz` — a timestamp **with time zone**; stored in UTC internally.
- `jsonb` — JSON stored in a parsed binary form; can be indexed and queried.
- `text[]` / `uuid[]` — a Postgres **array** column.
- `vector(768)` — a pgvector column holding 768 floating-point numbers (one embedding).
- `real` — a 4-byte float.
- "Nullable" = the column may be `NULL` (empty). "NOT NULL" = a value is required.

## Table: `stories`

**Purpose:** One row per **story cluster**. A cluster is a group of articles about the same event. A story starts life as a near-empty stub created during clustering (only `topic` + `source_count` are set — see `assign_cluster` in `engine/worldnews/db.py`), and is later filled in by the AI analysis step (`engine/worldnews/story_writer.py`, the `UPDATE stories SET ...` at lines 118–132).

| Column | Type | Nullable | Default | Meaning |
|--------|------|----------|---------|---------|
| `id` | `uuid` | NOT NULL | `gen_random_uuid()` | Primary key. Referenced by `articles.cluster_id` and listed inside `briefings.story_ids`. |
| `topic` | `text` | NOT NULL | — | Short title/topic of the story. Set at cluster creation, later overwritten by the analysis step with an "english_topic". |
| `first_seen` | `timestamptz` | NOT NULL | `now()` | When the cluster was first created. Not heavily used by the reader; `created_at` is what the web app filters on. |
| `source_count` | `int` | NOT NULL | `0` | How many articles are in the cluster, captured at creation time. The web app prefers the **live** count of joined `articles` and only falls back to this number (`db-datasource.ts`, `rowToStory`). |
| `lean_spread` | `jsonb` | NOT NULL | `'{}'::jsonb` | Bias spread, shaped `{"left":n,"center":n,"right":n}`. Counts how many of the story's sources lean left/center/right. Read by `toLeanSpread()` in `db-datasource.ts`. |
| `neutral_md` | `text` | nullable | — | The neutral, factual analysis in Markdown. **Also acts as the "is this story analyzed yet?" flag**: the web app's `rankedStories` and `storiesInRange` queries filter `WHERE neutral_md IS NOT NULL`, so a stub with no analysis is invisible to readers. |
| `beginner_md` | `text` | nullable | — | The "beginner" layer of analysis in Markdown (plain-language "what this means for you"). |
| `pro_md` | `text` | nullable | — | The "pro" layer in Markdown (economist/game-theory/market depth). |
| `sentiment` | `text` | nullable | — | Free-text sentiment. The web app coerces it to one of `bullish` / `neutral` / `bearish` (`toSentiment()`); anything unrecognized becomes `neutral`. |
| `impact_score` | `int` | nullable | — | 0–100 economic-impact score. The web app defaults a missing value to `50`. Used in ranking as `impact_score * region_relevance`. |
| `impact_summary` | `text` | nullable | — | One-sentence "why this matters to you" chain. |
| `affected_regions` | `text[]` | nullable | — | Array of region names this story affects, e.g. `{Indonesia,Global}`. The reader also tolerates a comma-separated string just in case (`rowToStory`). |
| `region_relevance` | `real` | nullable | — | 0–1 proximity to the home region (Indonesia). Defaults to `0.5` on read if null; treated as `0` inside the ranking SQL via `COALESCE(region_relevance, 0)`. |
| `translations` | `jsonb` | NOT NULL | `'{}'::jsonb` | Added by `0003`. Per-language copies of the generated text, keyed by language code, e.g. `{"id": {"topic": "...", "beginner_md": "...", ...}, "zh": {...}}`. English stays in the plain columns above. Written by `engine/worldnews/translate.py`; read by `tr()` in `db-datasource.ts`. |
| `created_at` | `timestamptz` | NOT NULL | `now()` | Creation timestamp. **This is the column the "today's stories" feed filters on** (`rankedStories` scopes `created_at` to the local day). |

**Constraints:** `PRIMARY KEY (id)`. No other unique constraints.

**Indexes:** None declared directly on `stories`. (Lookups by `id` use the primary-key index.)

## Table: `articles`

**Purpose:** One row per ingested news item. Created by `upsert_article` (`engine/worldnews/db.py`, lines 23–36); its embedding is filled in later by `set_article_embedding`; and clustering points it at a story via `cluster_id` (`assign_cluster`). The web app reads articles only to build each story's source list (`sourcesForStory` in `db-datasource.ts`).

| Column | Type | Nullable | Default | Meaning |
|--------|------|----------|---------|---------|
| `id` | `uuid` | NOT NULL | `gen_random_uuid()` | Primary key. |
| `url` | `text` | NOT NULL, **UNIQUE** | — | The article URL. The unique constraint is what makes ingestion idempotent: `INSERT ... ON CONFLICT (url) DO UPDATE SET title = EXCLUDED.title` (re-ingesting the same URL updates the title instead of creating a duplicate). |
| `title` | `text` | NOT NULL | — | Article headline. |
| `source` | `text` | NOT NULL | — | The outlet name, e.g. `Reuters`. Used as the **logical** link to `sources.name` (no real FK). |
| `country` | `text` | nullable | — | Country of the source/article, if known. |
| `published_at` | `timestamptz` | nullable | — | When the article was published. A story's display date is computed as `max(published_at)` across its articles (`storiesInRange`). |
| `fetched_at` | `timestamptz` | NOT NULL | `now()` | When we ingested it. |
| `summary` | `text` | nullable | — | Short summary text. |
| `embedding` | `vector(768)` | nullable | — | The 768-dim semantic embedding (from `nomic-embed-text`). Filled in after insert by `set_article_embedding`. Used for similarity clustering. **No vector index exists** (see [migrations.md](./migrations.md) gotcha). |
| `cluster_id` | `uuid` | nullable | — | Foreign key → `stories(id)`. Null until the article is assigned to a cluster. This is the **only real foreign key** in the schema. |
| `lean` | `text` | nullable | — | Per-article bias: `left` / `center` / `right`. Read by `toLean()` (anything unrecognized → `center`). |
| `lean_confidence` | `real` | nullable | — | Confidence (0–1) for the `lean` rating. |
| `author` | `text` | nullable | — | Added by `0002`. Article author, if known. |

**Constraints:**
- `PRIMARY KEY (id)`
- `UNIQUE (url)`
- `FOREIGN KEY (cluster_id) REFERENCES stories(id)` — declared inline. **No `ON DELETE` action**, so it defaults to `NO ACTION` (Postgres will refuse to delete a `stories` row while any article still points at it). See [relations.md](./relations.md).

**Indexes (from `0001_init.sql`):**
- `idx_articles_cluster` on `(cluster_id)` — speeds up `sourcesForStory` (`WHERE cluster_id = $1`) and the `articles a WHERE a.cluster_id = s.id` subqueries in `storiesInRange`.
- `idx_articles_published` on `(published_at)` — supports ordering/filtering by publish date.

## Table: `briefings`

**Purpose:** One row per **day**. The daily roundup that links that day's top stories. Written by `briefing_composer.py` (`INSERT INTO briefings ... ON CONFLICT (date) DO UPDATE ...`, lines 100–110), so re-running a day's pipeline overwrites that day's briefing rather than duplicating it. Read by `latestBriefing`, `briefingByDate`, `recentBriefings` in `db-datasource.ts`.

| Column | Type | Nullable | Default | Meaning |
|--------|------|----------|---------|---------|
| `id` | `uuid` | NOT NULL | `gen_random_uuid()` | Primary key. |
| `date` | `date` | NOT NULL, **UNIQUE** | — | The calendar day this briefing covers. Unique → at most one briefing per day; this is the `ON CONFLICT` target. node-pg parses this into a JS `Date` at local midnight; `toDateString()` in `db-datasource.ts` reads the local Y/M/D parts to avoid a UTC day-shift bug. |
| `story_ids` | `uuid[]` | NOT NULL | `'{}'` | Ordered array of `stories.id` values that this briefing covers. **Array, not a join table — no FK enforcement.** The web app turns these into the `storyIds` field and later fetches each story with `storiesByIds`. |
| `headline` | `text` | nullable | — | The briefing headline. |
| `overall_sentiment` | `text` | nullable | — | Day's overall sentiment; coerced like `stories.sentiment`. |
| `summary_md` | `text` | nullable | — | The briefing body in Markdown. **There is only one summary** (no beginner/pro split). The web app surfaces it in *both* the `beginnerMd` and `proMd` fields so the beginner↔pro toggle never renders blank (`rowToBriefing`). |
| `translations` | `jsonb` | NOT NULL | `'{}'::jsonb` | Added by `0003`. Per-language copies of `headline` + `summary_md`, keyed by language code. Written by `translate_briefing` in `translate.py`. |
| `created_at` | `timestamptz` | NOT NULL | `now()` | Creation timestamp. Used as a tiebreaker in ordering (`ORDER BY date DESC, created_at DESC`). |

**Constraints:** `PRIMARY KEY (id)`, `UNIQUE (date)`.

**Indexes:** None declared beyond the PK and the implicit unique index on `date`.

## Table: `sources`

**Purpose:** Long-lived **reputation memory** per news outlet. One row per outlet *name*. The bias analyst reads a source's history before analyzing, and `update_reputation` (`engine/worldnews/sources_memory.py`, lines 30–57) bumps the counters afterward. Used to track whether an outlet consistently leans a certain way.

| Column | Type | Nullable | Default | Meaning |
|--------|------|----------|---------|---------|
| `name` | `text` | NOT NULL | — | **Primary key.** The outlet name, e.g. `Reuters`. Matches `articles.source` logically (no FK). |
| `article_count` | `int` | NOT NULL | `0` | Total articles seen from this outlet. Incremented on each `update_reputation`. |
| `lean_left` | `int` | NOT NULL | `0` | Count of left-leaning articles seen. |
| `lean_center` | `int` | NOT NULL | `0` | Count of center articles seen. |
| `lean_right` | `int` | NOT NULL | `0` | Count of right-leaning articles seen. |
| `divergence_avg` | `real` | nullable | — | Running average of how much this outlet diverged from the consensus. `update_reputation` rolls it as a weighted mean over `article_count`. |
| `reliability` | `real` | nullable | — | A reliability score (0–1). Present in the schema; not written by `update_reputation` (which only touches counts + divergence). |
| `updated_at` | `timestamptz` | NOT NULL | `now()` | Last time this row was updated. Set to `now()` on every upsert. |

**Note on the lean counters:** `update_reputation` builds the column name dynamically (`lean_col = f"lean_{lean}"`) after validating `lean` is one of `left`/`center`/`right`. The value is whitelisted before being interpolated, so this is safe from SQL injection — but if you add a new lean value you must also add a matching `lean_*` column.

**Constraints:** `PRIMARY KEY (name)`. The `ON CONFLICT (name)` upsert relies on this PK.

**Indexes:** None beyond the primary-key index.
